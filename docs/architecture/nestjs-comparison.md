# Finvest API — Current Architecture vs NestJS

Comparison of the current Express + TSOA + Apollo stack versus an equivalent NestJS implementation,
assuming both expose REST and GraphQL endpoints backed by Prisma + PostgreSQL.

---

## Request flow comparison

### Current (Express + TSOA + Apollo)

```
HTTP Request
    │
    ├── REST → Express Router (TSOA-generated routes.ts)
    │              → expressAuthentication() [manual middleware]
    │              → Controller class (TSOA decorators)
    │              → Service
    │              → Repository
    │              → Prisma → PostgreSQL
    │
    └── GraphQL → Apollo Server (POST /graphql)
                   → buildApolloContext() [manual context builder]
                   → Resolver function
                   → Service
                   → Repository
                   → Prisma → PostgreSQL
```

### NestJS equivalent

```
HTTP Request
    │
    ├── REST → NestJS Router (auto-discovered from decorators)
    │              → Guards [e.g. JwtAuthGuard — applied via @UseGuards()]
    │              → Controller class (@Controller, @Get, @Post...)
    │              → Service (injected via DI)
    │              → Repository (injected via DI)
    │              → Prisma → PostgreSQL
    │
    └── GraphQL → Apollo Server (built-in @nestjs/graphql module)
                   → Guards [same JwtAuthGuard reused]
                   → Resolver class (@Resolver, @Query, @Mutation...)
                   → Service (injected via DI)
                   → Repository (injected via DI)
                   → Prisma → PostgreSQL
```

## Layer-by-layer differences

### Routing and controllers

| Aspect              | Current (TSOA)                                          | NestJS                                                  |
| ------------------- | ------------------------------------------------------- | ------------------------------------------------------- |
| Route registration  | `routes.ts` generated by TSOA at build time             | Auto-discovered at startup from decorators              |
| Controller style    | Class with `@Route`, `@Get`, `@Post` (TSOA decorators)  | Class with `@Controller`, `@Get`, `@Post` (NestJS decorators) |
| Swagger             | Generated from TSOA decorators → `swagger.json`         | Generated from `@nestjs/swagger` decorators             |
| Route file          | `src/routes/routes.ts` — do not edit manually           | No generated file — framework handles it at runtime     |

### GraphQL

| Aspect              | Current                                                 | NestJS                                                  |
| ------------------- | ------------------------------------------------------- | ------------------------------------------------------- |
| Setup               | Manual: `createApolloServer()` + `expressMiddleware()`  | `GraphQLModule.forRoot()` in the app module             |
| Schema              | SDL-first: `typeDefs` string in `schema.ts`             | Code-first (default): schema generated from decorators, or SDL-first optionally |
| Resolvers           | Plain objects/functions in `src/graphql/resolvers/`     | Classes with `@Resolver`, `@Query`, `@Mutation` decorators |
| Context / Auth      | `buildApolloContext()` — manual per-request function    | Same Guard system as REST — `@UseGuards(JwtAuthGuard)`  |
| Mounting            | Manually mounted before TSOA routes (order matters)     | Managed by the framework module system — no order issues |

The biggest practical difference here: in NestJS the same `JwtAuthGuard` works for both REST and
GraphQL. In the current stack, REST auth (`expressAuthentication`) and GraphQL auth
(`buildApolloContext`) are two separate implementations that must be kept in sync manually.

### Authentication

| Aspect              | Current                                                 | NestJS                                                  |
| ------------------- | ------------------------------------------------------- | ------------------------------------------------------- |
| REST                | `expressAuthentication()` — called by TSOA per `@Security('jwt')` | `JwtAuthGuard` — applied via `@UseGuards()` or globally |
| GraphQL             | `buildApolloContext()` — manual context builder         | Same `JwtAuthGuard` reused via `GqlExecutionContext`    |
| Public routes       | Default (no `@Security` decorator = public)             | `@Public()` decorator or skip the guard                 |
| Token payload       | `request.user` (REST) / `context.user` (GraphQL)        | `@CurrentUser()` custom decorator — same API for both  |

### Dependency injection

| Aspect              | Current                                                 | NestJS                                                  |
| ------------------- | ------------------------------------------------------- | ------------------------------------------------------- |
| DI system           | None — manual imports between layers                    | Built-in IoC container — services/repos declared as providers |
| Instantiation       | Static class methods (`UserService.getUsers()`)         | Instances injected via constructor (`constructor(private userService: UserService)`) |
| Testability         | Manual mocking of module imports in Jest                | Easy to swap providers in test modules (`useValue`, `useClass`) |

This is the most structural difference. NestJS manages object creation and lifetimes for you.
The current stack wires dependencies manually via imports, which works but makes testing and
swapping implementations more verbose.

### Module organization

| Aspect              | Current                                                 | NestJS                                                  |
| ------------------- | ------------------------------------------------------- | ------------------------------------------------------- |
| Structure           | Flat by type: `controllers/`, `services/`, `repositories/` | By feature module: `users/` contains controller + service + resolver + module |
| Boundaries          | Enforced by convention and code review                  | Enforced by the module system — a module only sees what it imports |
| Entry point wiring  | Manual in `src/index.ts`                                | `AppModule` declares and imports feature modules        |

---

## What stays the same

- **Services layer**: business logic lives in services in both architectures.
  NestJS services are just classes decorated with `@Injectable()` — the logic is identical.
- **Repositories layer**: the repository pattern works the same way.
  With `@nestjs/prisma` or a custom `PrismaService`, queries look identical.
- **Prisma + PostgreSQL**: no change at all.
- **REST vs GraphQL split**: same philosophy — REST for mutations with side effects, GraphQL for queries.

---

## Summary

The layers (controller → service → repository → DB) are the same in both architectures.
The main differences are about how the framework connects those layers:

| Topic               | Current stack                          | NestJS                                      |
| ------------------- | -------------------------------------- | ------------------------------------------- |
| Auth (REST+GQL)     | Two separate implementations           | One Guard, works for both                   |
| Dependency wiring   | Manual imports                         | IoC container                               |
| Module boundaries   | Convention                             | Enforced by module system                   |
| GraphQL setup       | Manual Apollo mounting                 | `GraphQLModule` handles it                  |
| Route registration  | Build-time code generation (TSOA)      | Runtime via decorators                      |
| Boilerplate         | Less upfront, more manual wiring       | More upfront modules, less manual wiring    |
| Learning curve      | Familiar if you know Express           | Steeper — Angular-like concepts (DI, modules) |
